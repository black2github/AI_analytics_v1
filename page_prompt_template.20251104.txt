Проведи детальный анализ каждой страницы отдельно по критериям, указанным ниже, в зависимости от типа требований страницы:

Для типа требований 'Функция' проверь:
- Указание назначения функции
- Если указаны шаги, то по всем шагам указано их назначение (цель шага) и на каждый шаг, кроме первого, есть переход
- Отсутствие неиспользованных входных параметров в описании алгоритма
- Определенность входных и выходных/исходящих параметров (названия, типы и форматы)
- Корректность используемых алгоритмов
- Полноту описания ситуаций или условий
- Обработку возможных ошибок, которые могут произойти или получены от других вызываемых функций или методов
- Указание возвращаемых ошибок для функций с выходными/исходящими параметрами
- Наличие логики для обработки ошибочных входных данных и сбоев инфраструктуры
- Соответствие типов и размерности используемых атрибутов размерностям и типам входных и исходящим параметров или размерностям и типам атрибутов используемых сущностей
- Обоснованность каждой операции над сущностью бизнес-целями
- Согласованность логику смены статусов используемой сущности с диаграммой состояний сущности
- Поддержку необходимых связей и атрибутов в используемой сущности
- Учет ограничений производительности на логическом уровне (например, сложность запросов к множеству связанных сущностей)
- Возможность реализация в рамках текущей архитектуры и интеграций
- Невозможность удаления требования без потери функциональности (например, является ли атрибут "цвет заказа" действительно нужным на логическом уровне)
- Отсутствие в условиях операций двусмысленностей в сравнениях или использованиях атрибутов
- Избегание расплывчатых фраз вроде "достаточно большой" или "в разумные сроки" в описании логики
- Наличие критериев успешности выполнения функции для каждого сценария
- Возможность смоделировать входные данные и ожидаемые результаты

Для типа требований 'Модель данных' проверь:
- Указание назначения сущности
- Указание, когда сущность создается и когда модифицируется
- Заполнение 'Описание поля' или 'Комментарий' для всех используемых атрибутов
- Что 'Описание поля' или 'Комментарий' исключает двойное толкование
- Заполнение всех полей, кроме полей с англоязычными кодами атрибутов/полей
- Описание связей с другими сущностями
- Указание типов данных для всех атрибутов
- Атрибуты типа Строка имеют либо ограничения длины и алфавита, либо имеют перечисляемый набор текстовых значений или ссылку на справочник
- Возможность поддержки указанных типов данных в СУБД
- Заполнение всех данных об уникальности и обязательности
- Указание значений по умолчанию для логических значений
- Отсутствие противоречий в атрибутах
- Отсутствие дублирующихся атрибутов

Для типа требований 'Процесс обработки' проверь выполнение:
- Все шаги пронумерованы
- Ясен следующий шаг после каждого действия или проверки
- На каждый шаг, кроме первого, есть переход
- События имеют наименование в форме краткого страдательного причастия прошедшего времени
- Исключающие шлюзы объединены с предшествующим им процессом

Для остальных типов требований или если тип требования не определен проверь:
- полноту
- непротиворечивость
- корректность
- осуществимость
- необходимость
- недвусмысленность
- проверяемость

ВАЖНО:
- Заголовок страницы и раздел "История изменений" анализировать не нужно.
- Для типа требований 'Модель данных' заполнять 'Техническое наименование поля' не требуется.
- Если по конкретному требованию по какому-то из критериев замечаний нет или оценка высокая, то информацию по данному критерию и требованию исключай из ответа.
