Сгенерируй OpenAPI-спецификацию версии 3.0.3 для каждой приложенной модели данных по описанию функции ниже. Для эндпоинтов используй следующие данные:

1. URL-путь сформируй исходя из наименования сущности модели данных (entity-name) и бизнес-смысла операции (operation-purpose) в формате:  /<entity-name>/<operation-purpose>. Например: /cards/get-card-list.
2. Если входной параметр один, имеет простой тип (нет сложных объектов/массивов), не содержит конфиденциальных данных и операция идемпотентна, то используй метод GET. Иначе используй метод POST.
3. Версия API: v2
4. Базовый URL сервера: https://api-gateway.sandbox.gboteam.ru/.
5. Для аутентификации используется Bearer token.
6. Идентификаторы имеют по умолчанию формат `UUID`.
7. При различных сценариях должны использоваться HTTP-коды 200 и 403: в случае ошибки авторизации или недостаточности прав возвращается код 403, в остальных случаях возвращается код 200. Для HTTP кода 200 блок с данными содержит объект со следующими полями:
    -`data`- опциональный объект с запрошенными полями/атрибутами документов (содержит массив документов docs), который отсутствует при наличии объекта с ошибкой.
    -`error`- опциональный объект с информацией об ошибке (отсутствует в случае успеха) содержащий:
    -- `code`- строковый код ошибки;
    -- `message`- строковое описание ошибки.
    -`traceId` - сквозной идентификатор цепочки операций, используемый для целей трассировки.
8. Для случая возврата набора данных/коллекции с поддержкой пагинации в ответе с данными укажи полное число доступных элементов коллекции (totalDocs), чтобы можно было рассчитать, сколько всего страниц может быть получено по запросу.
9. Сформируй идентификатор операции (operationId) исходя из бизнес-смысла операции и наименования сущности модели данных. Используй для идентификатора операции camelCase нотацию.
10. Описание функции начни с Англоязычного наименования (кода привилегии) с префиксом “Привилегии: ____“ и последующих двух переводов строк. Затем текст самого описания.
11. Для описания и summary используй русский язык. При необходимости вставить специальные символы используй двойные кавычки для всего набора строк описания или summary.
12. Сформируй примеры запроса, успешного ответа и ответа с ошибкой.
13. При формировании схем OpenAPI для полей, включенных в исходящие параметры функции полностью используй описания типов данных, указанные в столбце "Тип поля" модели данных, включая:
    - Для строк: максимальную длину (`maxLength`), если указана (например, "Строка (15)" означает `maxLength: 15`).
    - Для чисел: формат (например, "Число (16.2)" означает `format: double` с максимальной длиной 16 цифр, включая 2 десятичные), диапазоны значений (если указаны, например, "Положительное число или ноль").
    - Для дат: формат `date` или `date-time`, как указано в модели данных.
    - Для логических полей: тип `boolean`.
    - Для идентификаторов: формат `uuid`, если указано "GUID" или "UUID".
    - Другие ограничения, такие как допустимые символы или перечисления, если они указаны в описании поля или в столбце "ФЛК" (формально-логический контроль).
    - Если в столбце "ФЛК" или "Описание поля" указаны дополнительные ограничения (например, допустимые символы, диапазоны, форматы), они должны быть отражены в схеме OpenAPI (например, с использованием `pattern`, `minimum`, `maximum`, `enum`).
    - Формирование регулярных выражений (pattern):
    -- Анализируй название поля, тип данных и описание для определения необходимости паттерна. Если в описании поля указан конкретный формат (например, "ДД.ММ.ГГГГ"), создай соответствующий паттерн.
    -- Для кодов и идентификаторов (ИНН, КПП, БИК, счета, КБК и т.д.) формируй паттерны на основе:
    --- Указанной длины в типе данных;
    --- Семантики поля (если это код/номер, обычно цифры);
    --- Описания допустимых символов.
    -- Используй стандартные форматы для известных российских кодов и идентификаторов.
     - В случае указания ссылок на справочники формировать значения enum нужно с учетом справочников, упомянутых в описании полей.
14. Исключай из примеров опциональные поля, содержащие значения `null`.
